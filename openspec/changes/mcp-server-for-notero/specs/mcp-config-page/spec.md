## ADDED Requirements

### Requirement: MCP sidebar navigation item
The Organization sidebar SHALL include an "MCP Server" item placed directly under "Webhooks", with an appropriate icon, that navigates to `/organization/mcp` and highlights as active when that route is selected.

#### Scenario: Sidebar item visible
- **WHEN** an org member views the Organization sidebar
- **THEN** an "MCP Server" item is visible below the Webhooks item

#### Scenario: Active state on MCP route
- **WHEN** the user is on `/organization/mcp`
- **THEN** the MCP sidebar item renders in the active/highlighted state

### Requirement: MCP configuration page
The frontend SHALL expose a page at `/organization/mcp` with the following structure:
1. Page title: "MCP Server"
2. Description block explaining what the MCP server is and what AI clients can use it
3. Setup instruction block (static) explaining how to configure supported clients
4. Config JSON viewer block with syntax highlighting, monospace font, scrollable
5. Copy-to-clipboard button on the config block
6. API key management section (admin-only)

#### Scenario: Page loads for any org member
- **WHEN** any authenticated org member navigates to `/organization/mcp`
- **THEN** the page renders with title, description, instructions, and config block

#### Scenario: API key management hidden for non-admins
- **WHEN** a non-admin org member views the page
- **THEN** the API key management section is not rendered

### Requirement: Dynamic config JSON viewer
The config JSON block SHALL be fetched from `GET /organization/mcp/config` and rendered with syntax highlighting, monospace font, and a scrollbar when content overflows. The JSON shows both the `headers` approach and documents the `?email=` fallback in a comment or instruction block.

#### Scenario: Config rendered from API
- **WHEN** the page loads and the API call succeeds
- **THEN** the JSON block displays the current org's MCP configuration with the user's email pre-filled in `X-User-Email`

#### Scenario: Config shows placeholder when no API key
- **WHEN** `hasApiKey` is false in the API response
- **THEN** the JSON block shows placeholder values and a prompt to generate an API key

### Requirement: Copy-to-clipboard action
The config JSON block SHALL include a copy button that copies the full JSON string to the clipboard. The button SHALL cycle through states: idle → copied (with success feedback for 2 seconds) → idle. On failure, it SHALL show a failed state.

#### Scenario: Successful copy
- **WHEN** the user clicks the copy button
- **THEN** the full config JSON is written to the clipboard and the button shows a success state for 2 seconds before returning to idle

#### Scenario: Copy failure
- **WHEN** clipboard write fails (e.g., permission denied)
- **THEN** the button shows a failed state with a brief error message

### Requirement: Setup instructions block
The page SHALL include a static instruction block that explains how to connect supported MCP clients. It SHALL document:
- Where to paste the config (e.g., Claude Desktop's `claude_desktop_config.json`)
- The `X-User-Email` header approach for clients that support custom headers
- The `?email=` query parameter fallback for other clients
- That the API key is generated by an org admin

#### Scenario: Instructions rendered
- **WHEN** a user views the page
- **THEN** the instruction block is visible with complete setup steps for at least Claude Desktop

### Requirement: Admin API key management UI
For org admins, the page SHALL include an API key management section that allows:
- Viewing existing active keys (name, created date, last used date)
- Generating a new key (with name input) — shows the raw token once in a modal with a copy button and a warning that it won't be shown again
- Revoking an existing key (with confirmation dialog)

#### Scenario: Admin generates key
- **WHEN** an admin clicks "Generate API Key", enters a name, and confirms
- **THEN** a new key is created and the raw token is shown once in a modal with a copy button

#### Scenario: Token shown only once
- **WHEN** the admin closes the token modal
- **THEN** the raw token is no longer accessible from the UI; only the key name and metadata are shown

#### Scenario: Admin revokes key
- **WHEN** an admin clicks "Revoke" on an existing key and confirms the dialog
- **THEN** the key is removed from the list and subsequent MCP connections using that token are rejected
